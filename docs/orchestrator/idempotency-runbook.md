# Idempotency runbook

## What was implemented

Every `UserEvent` now carries a stable `event_id` (UUID) generated by the
producer at publish time.  The consumer records `(consumer_group, topic,
event_id)` in the `processed_events` table before executing any business
effect.  If the same triple arrives again, the handler logs a skip and
commits the Kafka offset — no business effect is repeated.

The Avro and Proto topics are independent.  Because each has its own consumer
group (`…-avro` / `…-proto`), the same `event_id` appears in two separate
rows and is processed by each flow exactly once.

## Table

```sql
CREATE TABLE processed_events (
    event_id        TEXT        NOT NULL,
    consumer_group  TEXT        NOT NULL,
    topic           TEXT        NOT NULL,
    partition       INTEGER     NOT NULL DEFAULT 0,
    offset          BIGINT      NOT NULL DEFAULT 0,
    processed_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    PRIMARY KEY (consumer_group, topic, event_id)
);
```

Dedup key: `(consumer_group, topic, event_id)`.  
`partition` and `offset` are observability columns only.

## Normal flow

```
1. producer   → uuid.New() → event.EventID set once
2. producer   → publishes to Avro topic  (with event_id in payload)
3. producer   → publishes to Proto topic (same event_id)
4. avro group → RecordIfNotExists → insert row → inner handler → commit offset
5. proto group→ RecordIfNotExists → insert row → inner handler → commit offset
```

After one user creation you should see two rows:

```sql
SELECT consumer_group, topic, event_id, processed_at
FROM   processed_events
ORDER  BY processed_at;
```

## Duplicate replay demo

### Crash-before-commit scenario

This is the realistic at-least-once scenario: the service crashes after the
business effect but before the Kafka offset is committed.  On restart the
broker re-delivers the same message.

```bash
# 1. Start the stack
docker compose up -d

# 2. Create a user and note the event_id in the logs (look for "kafka event received")
curl -s -X POST http://localhost:8080/api/v1/users \
  -H "Content-Type: application/json" \
  -d '{"name":"Alice","email":"alice@example.com","age":30}'

# 3. Verify two rows in processed_events
psql "$DATABASE_URL" -c "SELECT consumer_group, topic, event_id FROM processed_events;"

# 4. Simulate re-delivery: delete the processed_events rows so the guard is
#    bypassed, then reset the consumer group offsets to replay the message.
psql "$DATABASE_URL" -c "DELETE FROM processed_events;"

# Reset avro group offset (requires Kafka CLI tools or kafka-consumer-groups.sh)
kafka-consumer-groups.sh \
  --bootstrap-server localhost:9092 \
  --group user-service-avro \
  --reset-offsets --to-earliest \
  --topic com.br4.user.core.event.avro --execute

kafka-consumer-groups.sh \
  --bootstrap-server localhost:9092 \
  --group user-service-proto \
  --reset-offsets --to-earliest \
  --topic com.br4.user.core.event.proto --execute

# 5. Restart the service — consumers re-read the same messages.
docker compose restart app

# 6. Check that each event was processed once (two rows, same event_ids).
psql "$DATABASE_URL" -c "SELECT consumer_group, topic, event_id, processed_at FROM processed_events;"

# Expected: 2 rows, one per group — NOT 4 rows.
```

### Automated proof

The integration tests cover three cases without a running Kafka broker:

```bash
go test -tags=integration -v ./test/integration/... -run TestIdempotentHandler
```

| Test | What it proves |
|------|---------------|
| `TestIdempotentHandler_DeduplicatesOnSameEventID` | Inner handler called once even after two deliveries with the same event_id |
| `TestIdempotentHandler_IndependentPerTopic` | Same event_id on Avro topic and Proto topic each call inner once (two independent rows) |
| `TestIdempotentHandler_MissingEventID` | Pre-migration messages (empty event_id) fall through to inner without being recorded |

## Backward compatibility

Messages published before this change have `event_id = ""` (Avro default) or
`event_id` absent (Proto JSON).  `NewIdempotentHandler` detects the empty
string, logs a warning, and passes the message to the inner handler unchanged.
Idempotency is not guaranteed for such messages; they should be rare once all
producer instances are updated.

## Remaining gaps (future steps)

- DLQ: a poison-pill message (bad payload) still blocks the session until restart.
- Outbox: the producer's `publishEvent` is still fire-and-forget; a crash between
  the DB write and the Kafka publish can lose the event entirely.
- Cleanup policy: `processed_events` grows indefinitely; a TTL or partition strategy
  is needed in production.
